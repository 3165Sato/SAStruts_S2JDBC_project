# AI生成テストコード評価レポート

## 1. 検証背景

レガシーJava環境では、CI/CDの導入やAIによるテストコード生成の活用を検討する場面がある。一方で、実務を想定したサンプルにはSeasar2 / SAStruts / S2JDBC / S2JUnit4のようなレガシーJava構成が残っていることがあり、最新のSpring BootやJUnit 5を前提とした検証だけでは判断材料として不十分である。

また、S2JUnit4のテストではExcelテストデータに依存しているケースがあり、保守性、レビュー性、AIによる生成・修正との相性に課題がある。Excel形式は人間が見るには便利な面がある一方で、Git差分でのレビューや、AIが構造を理解して安全に編集する用途では扱いにくい。

そのため、本検証ではDoltengで生成した Seasar2 / SAStruts / S2JDBC 構成の個人検証環境を用意し、AI生成テスト、S2JUnit4のCI実行、DBありテスト、Excelなしテストデータ準備を段階的に確認した。

## 2. 検証環境

| 項目 | 内容 |
| --- | --- |
| Framework | Seasar2 / SAStruts / S2JDBC |
| Project | Dolteng生成プロジェクト |
| Test Framework | S2JUnit4 |
| Java | プロジェクト指定は Java 1.7、実行環境は JDK 8 |
| Maven | 3.6.3 |
| CI | GitLab CI |
| DB | H2インメモリDB |
| DB Access | S2JDBC / JdbcManager |

補足として、Maven 3.9系ではHTTP repositoryブロックによりSeasar系リポジトリで問題が発生する。そのため、本検証ではJDK 8 + Maven 3.6.3の組み合わせを採用した。

## 4. AIによるS2JUnit4テスト生成の評価

`@RunWith(Seasar2.class)` を使用したS2JUnit4テストをAIが生成できることを確認した。特に、請求書ステータス管理のような状態遷移ロジックに対して、正常系、異常系、境界値、再実行不可などの観点をテストとして追加できた。

また、単一Serviceから複数Logic / Store相当クラスへ責務分割した後も、既存テストを維持しつつテスト範囲を拡張できた。これは、AIに対して業務ルール、制約、対象クラス、変更してはいけないファイルを明示したことが有効だったと考えられる。

ただし、実業務レベルでは依存関係がより複雑になる。複数Service、DAO、外部インターフェース、トランザクション境界が絡む場合、AIに丸投げするだけではテスト観点の漏れや不適切な前提設定が起きる可能性がある。そのため、人間がテスト観点、対象範囲、前提条件、禁止事項を明示したうえで、AIを実装補助として使う形が現実的である。

## 5. CI/CD実現性の評価

ローカル環境では `mvn clean test` が成功し、同じコマンドをGitLab CI上でも実行できた。S2JUnit4のテストは、JDK 8 + Maven 3.6.3 の組み合わせで安定して実行できた。

また、意図的に業務ロジックへバグを混入した場合、S2JUnit4テストが失敗し、GitLab CIのパイプライン失敗として検知できた。この結果から、AI生成テストをCIに載せることで、レガシーJava環境でも一定の品質ゲートとして活用できる見込みがある。

重要な注意点として、Maven 3.9系ではHTTP repositoryブロックによりSeasar系リポジトリの解決で問題が出る。そのため、レガシーJava環境をCI化する場合は、Javaバージョン、Mavenバージョン、依存リポジトリを固定し、再現性を担保することが重要である。

## 6. DBありS2JUnit4テストの評価

既存のインメモリ実装である `InvoiceService` は変更せず、DB検証用の最小機能として `DbInvoiceService` を追加した。これにより、既存機能をDB化するリスクを避けながら、S2JDBCによるDBアクセスのテストだけを切り出して検証できた。

DBにはH2インメモリDBを使用し、`src/test/resources/jdbc.dicon` によりテスト用DB設定を追加した。main側の `jdbc.dicon` は変更していない。

S2JDBC / `JdbcManager` を使用して、以下を検証した。

- DBへの登録
- IDによる検索
- ステータス更新
- 登録件数取得

GitLab CI上でもDBありテストは成功した。H2 TCPサーバ方式ではなくインメモリ方式を採用したことで、CI上の起動待ち、ポート競合、サーバ停止漏れといった不安定要因を避けられた。

## 6.1 複数TABLE Scenario Fixture検証

単一テーブルだけでなく、複数TABLEをまたぐ前提データをExcelなしで作成できることを確認した。

この検証では、顧客、部署、請求書、承認履歴などの複数テーブルを組み合わせた業務シナリオをScenario Fixtureで作成した。例えば「承認済み請求書シナリオ」では、顧客・部署・請求書・承認履歴をまとめて作成できる。

Builder / Fixture / SQLファイルを組み合わせることで、複数TABLEのテストデータ準備が可能であることを確認した。Scenario Fixtureは、複数テーブルにまたがる「業務状態」を作る役割である。

この検証もGitLab CIで成功しており、Excelに依存しない複数TABLE前提データ作成の実現性を確認できた。

## 6.2 複数Service / 複数Logic / DBアクセスServiceを跨ぐ業務シナリオ検証

実業務に近い構成として、請求書支払確定処理を追加し、複数Service / 複数Logic / DBアクセスServiceを跨ぐ業務シナリオをS2JUnit4で検証した。

この検証では、`InvoicePaymentConfirmService` を業務処理の入口とし、以下の呼び出し関係で処理を組み立てた。

- `InvoicePaymentConfirmService`: 支払確定処理全体の入口
- `InvoicePaymentConfirmValidationLogic`: 支払確定可否の判定
- `InvoicePaymentHistoryLogic`: 支払確定履歴Entityの作成
- `DbScenarioInvoiceService`: 請求書の取得とステータス更新
- `DbApprovalHistoryService`: 履歴の登録・検索

S2JUnit4テストでは、以下の観点を確認した。

- 承認済み請求書を支払確定できる
- 支払確定後に請求書ステータスが `PAYMENT_CONFIRMED` になる
- 支払確定履歴が登録される
- 支払確定履歴の `INVOICE_ID` が対象請求書IDと一致する
- 支払確定履歴の `STATUS` が `PAYMENT_CONFIRMED` である
- 未承認・差戻し済み・支払確定済み・存在しないIDではエラーになる
- 異常時に請求書ステータスや履歴件数が変わらない

この検証により、AI生成テストでも複数クラスを跨ぐ業務処理を検証できることを確認した。一方で、副作用検証は明示的に指示しないと漏れやすい。特に、異常時にDB状態が変わらないこと、履歴が二重登録されないこと、履歴の `INVOICE_ID` と `STATUS` が正しいことは、人間がテスト観点として明示する必要がある。

また、Scenario Fixtureにより複数TABLEの前提データを準備しやすくなり、承認済み請求書や差戻し済み請求書などの業務状態をテスト開始時点で表現できることも確認した。

## 6.3 トランザクション・ロールバック検証

複数DB更新を伴う業務Serviceで、途中失敗時にロールバックされるかを検証した。

対象は `InvoicePaymentConfirmService#confirmPayment(Long invoiceId)` とした。この処理では、支払確定時に `DB_SCENARIO_INVOICE` の請求書ステータスを `PAYMENT_CONFIRMED` に更新し、続けて `DB_APPROVAL_HISTORY` に支払確定履歴を登録する。

検証では、承認済み請求書を前提に、履歴登録時に `RuntimeException` を発生させた。本番コードには失敗注入用のフラグや分岐を追加せず、S2JUnit4テスト側で `DbApprovalHistoryService` の失敗版へ一時的に差し替えることで、テスト専用の失敗を注入した。

例外発生後、以下を確認した。

- 請求書ステータスが `PAYMENT_CONFIRMED` にならず、`APPROVED` のままであること
- 支払確定履歴の件数が増えていないこと
- ローカルの `mvn clean test` が成功すること
- GitLab CIでも成功すること

この結果により、Service層のトランザクション境界により、請求書ステータス更新と履歴登録が同一トランザクションとして扱われ、途中失敗時にまとめてロールバックされることを確認できた。

### トランザクション境界の評価

今回の構成では、業務処理の入口であるService層にトランザクション境界を置き、Logic層にはトランザクション境界を置かない方針とした。

- Service層: 業務入口、処理フロー制御、DBアクセスServiceの呼び出しを担当し、トランザクション境界を持つ
- Logic層: 入力検証、状態遷移判定、履歴Entity生成などを担当し、トランザクション境界は持たない
- DBアクセスService: S2JDBC / `JdbcManager` による登録、検索、更新を担当し、業務Serviceから呼ばれる場合は外側のトランザクションに参加する

この分担により、トランザクション境界を業務入口に集約しつつ、Logic層を判定やEntity生成に集中させる構成を確認できた。実務適用時にも、どのServiceを業務トランザクションの入口にするかを人間が明確に設計することが重要である。

また、S2JUnit4ではテストメソッド自体がトランザクション制御を持つ場合があるため、Service層のトランザクション境界を直接検証したい場合は、テスト側のトランザクション挙動にも注意が必要である。

## 7. Excelなしテストデータ準備方式の比較

### Javaコード直接方式

Javaコード上でEntityを直接生成し、必要な項目をセットする方式である。

最小検証には向いており、すぐに実装できる。一方で、テストケースが増えると同じ初期化処理が重複しやすく、項目追加時の修正範囲も広がりやすい。

### TestDataBuilder方式

`DbInvoiceTestDataBuilder` で `DbInvoice` オブジェクトを作成する方式である。

`unapprovedInvoice` や `approvedInvoice` のように、テストデータの意味をメソッド名で表現できる。項目追加時の修正範囲をBuilder内に寄せやすく、AIがテストコードを生成する際にも意図を読み取りやすい。

### Fixture方式

`DbInvoiceFixture` でDB登録済みの状態を作成する方式である。

Builderで作成したデータをDBへinsertし、テスト開始時点の業務状態を作る。業務シナリオに近いテスト準備に向いており、ServiceやLogicの振る舞いを確認するテストで有効である。

### SQLファイル方式

DDLや固定初期データをSQLファイルで管理する方式である。

Git差分でレビューしやすく、複数TABLEや固定マスタの準備に向いている。ただし、テストケースごとの細かな業務状態をすべてSQLで表現するとメンテナンスが重くなる可能性がある。そのため、Builder / Fixtureの代替ではなく補完として使うのがよい。

### Scenario Fixture方式

複数TABLEをまたぐ業務状態をまとめて作成する方式である。

顧客、部署、請求書、承認履歴のように、単一テーブルでは表現できない業務シナリオの前提データを準備する場合に有効である。Builderで1件単位のEntityを作成し、必要に応じてFixtureやServiceを使ってDBへ登録することで、テスト開始時点の業務状態を明確にできる。

## 8. Builder / Fixture / Scenario Fixture / SQLの使い分け

本検証で確認した使い分けは以下である。

- Builder: 1件のEntityやDTOを作る
- Fixture: 1テーブルまたは1機能単位で、DB登録済み状態を作る
- Scenario Fixture: 複数TABLEをまたぐ業務シナリオ状態を作る
- SQL: DDL、固定マスタ、複数テーブルの基本初期状態を作る

実務では、以下の組み合わせが現実的である。

- DDL・固定マスタ: SQLファイル
- 業務シナリオごとのデータ: Scenario Fixture
- 1機能単位のDB登録済みデータ: Fixture
- 個別EntityやDTOの生成: Builder
- テスト固有の微調整: Builderの上書きメソッド

このように役割を分けることで、Excelテストデータへの依存を減らしつつ、レビューしやすさ、AIによる生成しやすさ、保守性を両立しやすくなる。

## 9. 今回の検証で分かったこと

- Seasar2 / S2JUnit4でもAI生成テストは一定活用できる
- AIは明確な業務ルールや状態遷移表があるとテストを生成しやすい
- CIに載せることで、AI生成テストを品質ゲートとして使える
- Excel依存を減らすには、Builder / Fixture / Scenario Fixture / SQLファイル方式が有効である
- 複数TABLEをまたぐ前提データも、Scenario FixtureによりJavaコード上で表現できる
- AI生成テストでも複数Service / 複数Logic / DBアクセスServiceを跨ぐ業務処理を検証できる
- 副作用検証は明示的に指示しないと漏れやすい
- Service層にトランザクション境界を置くことで、複数DB更新の途中失敗時にロールバックできることを確認できた
- 本番コードを汚さず、テスト側の差し替えにより失敗注入できることを確認できた
- 実務適用では、人間が業務ルール・副作用・DB状態を整理してからAIに依頼する必要がある
- AIに丸投げするのではなく、人間がテスト観点と制約を明示する必要がある
- 実業務適用には複数TABLE、複数Service、外部依存、トランザクションなどの追加検証が必要である

## 10. 注意点・リスク

- 今回のロジックは実業務より単純化されている
- Level 2〜4では、Logic / Store相当クラスは必ずしもS2コンテナ管理ではない
- 実業務では複数Service / 複数Logic / DAO / 外部IFが絡むため、AI生成テストの精度は下がる可能性がある
- AIはテスト観点を漏らす可能性があるため、人間によるレビューは必須である
- Excel脱却は一括置換ではなく、対象テストの性質に応じて段階的に進めるべきである
- Scenario Fixtureが複雑化すると、Fixture自体の保守性が課題になる可能性がある

## 12. 総括

今回の検証により、Seasar2 / S2JUnit4のようなレガシー環境でも、AI生成テストとCI/CDの組み合わせには十分な検証価値があることを確認できた。

特に、状態遷移ロジックに対する正常系・異常系テスト、GitLab CIによる自動実行、H2 + S2JDBCによるDBありテスト、Excelなしテストデータ準備、複数TABLEをまたぐScenario Fixture、複数Service / 複数Logic / DBアクセスServiceを跨ぐ業務シナリオ、トランザクション・ロールバック検証について、一定の実現性を確認できた。

一方で、実業務の複雑性を考えると、AIに完全自動化を期待するのではなく、人間が仕様、テスト観点、制約、前提条件を整理し、AIに実装補助させる形が現実的である。

次の段階では、複数TABLE・複数Serviceをまたぐ業務シナリオで追加検証し、実業務適用時の課題をより具体化する必要がある。
