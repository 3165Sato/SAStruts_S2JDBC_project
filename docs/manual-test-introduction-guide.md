# S2JUnit4テスト導入手順

## 1. 目的

このドキュメントは、実ソースコードに対してS2JUnit4テストを導入するための手順書である。

最初からAI生成テストを使うのではなく、まず人力で最小のS2JUnit4テストを追加し、既存プロジェクトでテスト追加、ローカル実行、CI実行が成立するかを確認する。

小さな成功パターンを作った後に、AIによるテスト観点整理やテストコード生成へ段階的に進む。既存機能、既存テスト、dicon構成、DB設定を壊さず、安全に始めることを重視する。

## 2. 全体方針

導入は以下の順序で進める。

1. 現状構成を把握する
2. 既存テストをローカルで実行する
3. 既存テストをCIで実行する
4. 小さい対象機能を選ぶ
5. 人力でS2JUnit4テストを1〜3本追加する
6. ローカルで確認する
7. CIで確認する
8. DBありテストが必要なら最小構成で確認する
9. Excelテストデータ依存がある場合は代替方式を検討する
10. 成功パターンを整理してからAI活用へ進む

失敗時に原因を切り分けられるよう、工程を飛ばさず段階的に進める。AIは最初は解析補助、観点整理補助として使い、コード生成は後段にする。

## 3. Phase 0: 事前確認

最初に、対象プロジェクトの構成を確認する。

- Javaバージョン
- Maven / Ant の利用有無
- Seasar2 / SAStruts / S2JDBC / S2JUnit4 の利用有無
- dicon構成
- web.xml構成
- 既存テストの配置
- 既存テストの実行方法
- Excelテストデータの有無
- DB接続方式
- CI設定の有無
- 変更してはいけないファイル
- 本番DBや実データに接続しないこと

特に、`jdbc.dicon`、`app.dicon`、`s2jdbc.dicon`、`s2junit4.dicon`、`web.xml` は既存構成への影響が大きいため、安易に変更しない。

## 4. Phase 1: 既存テストのローカル実行

まず、既存テストをローカルで実行する。

既存テストが失敗している状態で、新規テストを追加しない。失敗原因が既存環境なのか、新規テストなのか切り分けにくくなるためである。

Mavenプロジェクトの場合の実行例:

```bash
mvn clean test
```

Antプロジェクトの場合は、既存プロジェクトで定義されているテスト実行ターゲットに従う。

確認すること:

- JDKバージョンが既存プロジェクトの指定と合っている
- Maven / Ant のバージョンが既存環境と合っている
- Seasar系の古い依存関係が解決できる
- HTTPリポジトリが必要な場合、Maven 3.9系のHTTPブロックに注意する
- 実行手順、必要な環境変数、注意点をメモする

## 5. Phase 2: 既存テストのCI実行

ローカルで成功した既存テストをCIでも実行する。

確認すること:

- CIでJDK / Maven / Ant のバージョンを固定している
- ローカルとCIで同じテストコマンドを実行できる
- CIでDB接続が必要か確認している
- CIでExcelテストデータが参照できる
- CI上でファイルパス、文字コード、タイムゾーン差分が問題にならない

CIが成功するまでは、AI生成テストには進まない。まず、既存テストを品質ゲートとして実行できる状態を作る。

## 6. Phase 3: 最初の対象機能選定

最初に選ぶ対象は、小さく、依存関係が少なく、仕様が明確なものにする。

おすすめ対象:

- 入力チェックLogic
- 状態判定Logic
- DBアクセスのない小さいService
- 依存関係が少ない処理
- 業務ルールが明確な処理

避ける対象:

- 複数外部システムと連携する処理
- 仕様が曖昧な処理
- 副作用が多すぎる処理
- 既存テストが不安定な処理
- DB、ファイル、外部APIが複雑に絡む処理

最初の目的は、網羅的なテストを作ることではなく、対象プロジェクトでS2JUnit4テストを安全に追加できることを確認することである。

## 7. Phase 4: 人力でS2JUnit4テストを追加する

最初はAIにテストコードを書かせない。人間が既存テストの書き方に合わせて、1〜3本だけ追加する。

守ること:

- S2JUnit4の形式を維持する
- 必要に応じて `@RunWith(Seasar2.class)` を使用する
- Serviceを `new` せず、必要に応じてS2コンテナからDIする
- 既存のdiconを不用意に変更しない
- 正常系1本、異常系1本程度から始める
- テスト名から検証内容が分かるようにする

追加するテストの例:

- `null` 入力時の例外
- 不正ステータス時の例外
- 正常な状態遷移
- 既存メソッドの戻り値確認

まずは既存テストの作法に合わせる。新しいテスト設計や大きなリファクタリングは、最初の成功後に検討する。

## 8. Phase 5: ローカル・CI確認

追加したテストをローカルで実行する。

確認すること:

- 新規テストが成功する
- 既存テストも含めて成功する
- テスト順序に依存していない
- 既存diconやDB設定を壊していない

ローカルで成功したら、CIでも確認する。

失敗した場合は、以下を切り分ける。

- 新規テストの実装問題
- テスト対象コードの仕様理解不足
- ローカルとCIの環境差分
- 既存テストの不安定性
- DB、ファイル、Excelテストデータなどの参照問題

## 9. Phase 6: DBありテストを検討する

DBありテストは、必要になった場合のみ進める。最初からDBありテストにすると、切り分け対象が増えるためである。

確認項目:

- テスト用DBを使えるか
- 本番DBに接続していないか
- H2などのインメモリDBへ切り替え可能か
- テスト専用 `jdbc.dicon` を用意できるか
- テストごとにDB初期化できるか
- CIでDBありテストが安定実行できるか

最初は、登録、検索、更新、件数取得など、小さいDBアクセステストに限定する。

## 10. Phase 7: Excelテストデータ依存の確認

既存テストがExcelデータに依存しているか確認する。

Excelテストデータが存在しても、いきなり全廃しない。まず用途を分類し、代替しやすいものから段階的に検討する。

分類例:

- DDL
- 固定マスタ
- 業務シナリオデータ
- 1件単位のEntityデータ
- 期待値データ

代替候補:

- DDL / 固定マスタ: SQLファイル
- 1件単位のEntity: TestDataBuilder
- DB登録済み状態: Fixture
- 複数テーブルの業務状態: Scenario Fixture

Excel形式が向いているケースもあるため、置き換えは一括ではなく、テストの性質に応じて判断する。

## 11. Phase 8: 成功パターンの整理

人力で小さく成功したら、成功パターンを整理する。

記録すること:

- テスト対象クラス
- 追加したテストケース
- テスト実行方法
- 必要なdicon
- DB初期化方法
- テストデータ準備方法
- CI設定
- 注意点
- 既存テストへの影響

この記録を、後続のAI活用時の参照例にする。AIに依頼する際も、この成功パターンを前提として提示することで、既存構成に沿った提案やコード生成をしやすくなる。

## 12. Phase 9: AI活用に進む条件

AIを使うのは、以下を満たしてからにする。

- 既存テストがローカルで成功している
- 既存テストがCIで成功している
- 人力で追加したS2JUnit4テストが成功している
- 対象機能の構成が把握できている
- テスト配置、命名、データ準備方法が整理されている

これらを満たしていない状態でAI生成テストを追加すると、失敗原因がAI生成結果なのか、環境なのか、既存構成なのか判断しにくくなる。

## 13. Phase 10: AIの使い方

AIの最初の使い方は、コード生成ではなく、解析と観点整理にする。

最初に依頼する内容:

- 対象機能の解析
- テスト観点の洗い出し
- 異常系の候補出し
- 副作用検証の候補出し
- 既存テストの不足観点レビュー

その後、限定的にテストコード生成へ進む。

コード生成を依頼する場合も、以下を明示する。

- 対象クラス
- 既存テストの作法
- 使用するテストフレームワーク
- Javaバージョン
- 変更してよいファイル
- 変更してはいけないファイル
- DBやExcelテストデータの扱い

## 14. 注意点

- 既存テストが不安定な状態でAI生成テストを追加しない
- 本番DBや実データに接続しない
- diconを不用意に変更しない
- JUnit 5やSpring Bootへ寄せない
- Javaバージョンに合わない構文を使わない
- AI生成結果は必ず人間がレビューする
- CIで必ず検証する
- ローカルだけの成功で完了扱いにしない

## 15. 最初の完了基準

最初の導入完了基準は以下とする。

- 実ソース上で人力追加したS2JUnit4テストが1本以上存在する
- ローカルでテストが成功している
- CIでテストが成功している
- 既存テストを壊していない
- テスト追加手順がメモされている
- 次にAI活用へ進める状態になっている

この状態になれば、AIによるテスト観点整理や限定的なテストコード生成へ進む準備ができている。

## 16. まとめ

実ソースへの導入では、最初からAI生成テストを使わない。

まずは人力で最小のS2JUnit4テストを追加し、対象プロジェクトでテスト追加、ローカル実行、CI実行が成立することを確認する。

成功パターンができてからAIを使うことで、AIの効果と失敗原因を切り分けやすくなる。AIは最初から自動化する道具ではなく、テスト観点整理とコード生成補助として段階的に使うのが現実的である。
