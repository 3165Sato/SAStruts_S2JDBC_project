# AI生成テストコード レビューチェックリスト

## 1. 目的

AI生成テストは、テストコード作成の効率化に有効である。一方で、生成結果をそのまま信用しすぎることは危険である。AIは業務仕様を誤って補完したり、異常系や副作用の確認を漏らしたり、実行環境に合わない構文やフレームワークを使う可能性がある。

そのため、AIが生成したテストコードを実務で利用する前に、人間が業務仕様、テスト観点、副作用、DB状態、CI実行可否をレビューする必要がある。

このチェックリストは、Seasar2 / SAStruts / S2JDBC / S2JUnit4 のようなレガシーJava環境で、AI生成テストを実務に適用する前の確認観点を整理するものである。

## 2. 全体チェック

- [ ] テスト対象クラスが正しい
- [ ] テスト対象範囲が明確である
- [ ] 正常系だけでなく異常系も含まれている
- [ ] 境界値テストが含まれている
- [ ] 業務ルールがテスト名・テスト内容から読み取れる
- [ ] テストが実装詳細に依存しすぎていない
- [ ] テストが単にカバレッジを増やすだけになっていない
- [ ] `mvn clean test` で実行できる
- [ ] GitLab CIで実行できる

## 3. S2JUnit4固有のチェック

- [ ] `@RunWith(Seasar2.class)` を使用している
- [ ] S2コンテナからDIすべきServiceを `new` していない
- [ ] `@PostBindFields` など、初期化タイミングが適切である
- [ ] dicon構成を不要に変更していない
- [ ] `app.dicon` / `jdbc.dicon` / `s2junit4.dicon` の読み込みが意図通りである
- [ ] S2JUnit4のテストとして成立している
- [ ] 通常のJUnit4テストに置き換わっていない
- [ ] Spring BootやJUnit 5の前提に寄っていない
- [ ] Java 1.7互換の構文で書かれている

## 4. 業務ロジックテストのチェック

- [ ] 状態遷移の正常系がある
- [ ] 状態遷移の不正パターンがある
- [ ] 異常系で例外が発生することを確認している
- [ ] 異常系で状態が変わっていないことを確認している
- [ ] 入力値チェックが網羅されている
- [ ] `null`、空文字、0、マイナス、上限値などの境界が確認されている
- [ ] 複数Service / 複数Logicを跨ぐ場合、呼び出し順や責務分担がテストで確認されている
- [ ] 業務仕様にない期待値をAIが勝手に作っていない
- [ ] テスト名が業務ルールを表している
- [ ] テスト対象外の仕様まで過剰に検証していない

## 5. DBありテストのチェック

- [ ] H2などテスト用DBで安定実行できる
- [ ] テストごとにDB状態が初期化されている
- [ ] テスト順序に依存していない
- [ ] テストデータの登録・検索・更新・削除が適切に確認されている
- [ ] DBに登録したデータとテスト期待値が対応している
- [ ] トランザクションやロールバックの扱いが明確である
- [ ] main側の `jdbc.dicon` を不用意に変更していない
- [ ] CI上でDB起動待ちやポート競合が起きない構成である
- [ ] テスト失敗時にDB状態の原因を追いやすい

## 6. Excelなしテストデータ準備のチェック

- [ ] Excelに依存していない
- [ ] TestDataBuilderの役割がデータ作成に限定されている
- [ ] Fixtureの役割がDB登録済み状態の準備に整理されている
- [ ] SQLファイルはDDLや固定マスタに使われている
- [ ] Builder / Fixture / SQLの責務が混ざっていない
- [ ] テストデータの意図がメソッド名やSQLファイル名から分かる
- [ ] テストデータの重複が過剰でない
- [ ] 項目追加時に修正箇所が局所化されている
- [ ] テスト固有の値変更はBuilderの上書きメソッドなどで表現されている

## 7. AI生成テストでありがちな問題

- 正常系しか作らない
- 例外が出ることだけ確認し、副作用を確認しない
- `assert` が弱い
- テスト名が曖昧で、何を検証しているか分かりにくい
- 実装に合わせすぎて仕様の検証になっていない
- 存在しない仕様を勝手に補完する
- モックを使いすぎる
- DB初期化が不十分である
- テスト順序に依存する
- Javaバージョンに合わない構文を使う
- JUnit 5やSpring Bootに寄せてしまう
- 既存のdicon / web.xml / WTP構成を不要に変更する

## 8. レビュー時のOK/NG例

### OK例

- 業務ルールがテスト名から分かる
- 正常系・異常系・境界値がある
- S2コンテナからServiceをDIしている
- テストデータ準備がBuilder / Fixtureで整理されている
- 例外発生時に状態が変わっていないことも確認している
- DBありテストで初期化処理が明示されている

### NG例

- Serviceを `new` している
- `assert` がない、または弱い
- 例外だけ確認して状態不変性を見ていない
- ExcelやDB状態に暗黙依存している
- テストの前提データが読み取れない
- テスト実行順序に依存している
- AIが作った期待値の根拠が仕様から確認できない

## 9. 人間が重点的に見るべきポイント

- 業務仕様とテスト観点が一致しているか
- AIが仕様を勝手に補完していないか
- 異常系と境界値が十分か
- DB状態や副作用が検証されているか
- CIで安定して実行できるか
- テストが保守可能な形になっているか
- 既存のSeasar2 / S2JUnit4構成を壊していないか
- テストデータ準備方式が対象テストの性質に合っているか

## 10. まとめ

AI生成テストは、S2JUnit4のようなレガシー環境でも有効に使える可能性がある。特に、業務ルールや状態遷移が明確であれば、正常系・異常系・境界値を含むテストを効率よく生成できる。

ただし、AI生成結果をそのまま採用するのではなく、人間がレビューすることが前提である。特に、業務ルール、状態遷移、DB状態、副作用、テストデータ準備方式は重点的に確認する必要がある。

このチェックリストを使うことで、AI生成テストをCI/CDの品質ゲートとして活用しやすくなる。AIはテスト作成の補助として使い、人間が仕様と品質を確認する体制を組み合わせることが重要である。
