# 単体テストコード作成方針書

## 1. はじめに

### 背景

既存のJava/Seasar2アプリケーションは、テストがほとんど整備されていない。そのため品質確保やリファクタリングのたびに、目視と手作業の確認に頼らざるを得ないのが実情だ。加えて今後はAIエージェントによるテストコード生成も取り入れたく、その土台として共通の方針を定める。

### 目的

本方針書は、次の3者が同じ前提でテストに向き合えるようにすることを狙いとする。

- **開発チーム**：担当者によって品質がばらつかないよう、作成・保守のルールを揃える。
- **上長・レビュー担当者**：導入のねらいと効果を共有し、工数をかける判断の材料にする。
- **AIエージェント**：人間と同じ方針で生成できるよう、作業指示を標準化する。

## 2. 技術前提

方針は以下の技術スタックを前提とする。

| 項目 | 内容 |
| --- | --- |
| Javaバージョン | Java 8 |
| フレームワーク | Seasar2 |
| テストフレームワーク | S2JUnit（JUnit4形式で `@RunWith(Seasar2.class)` を使用） |
| DB | H2インメモリDB。Daoクラス経由でアクセスする |
| モックライブラリ | EasyMockに統一 |
| assert | `assertEquals`、`assertTrue`、`assertFalse`、`assertNull` などJUnit標準のメソッドを使用 |
| Excel方式 | 使用しない |

## 3. テスト対象方針

### 3-1. レイヤー別対象

- **Service層**：トランザクション境界であり、業務処理の入口。複数Logicの連携やDB更新結果が意図どおりかを検証する。外部依存はEasyMockで差し替え、DBを伴う処理はH2インメモリDBで確認する。
- **Logic層**：業務ロジックの中心。判定・状態遷移・計算を重点的にテストする。Daoや外部依存は原則EasyMockに差し替え、分岐・境界値・例外条件を見る。ただしDaoを切り離すと検証しづらい構造の場合は、H2を併用してよい。
- **Dao層**：DBアクセスを担う層。H2インメモリDBを使い、SQL・ORマッピング・検索条件・戻り値・更新件数の正しさを検証する。

### 3-2. 優先順位

次のような処理から優先的に着手する。

- 業務影響・障害影響が大きい処理
- ステータス遷移や金額計算を伴う処理
- DB更新を伴う処理
- 境界値・例外条件が多い処理
- 過去に不具合が出た処理、AI生成コードの影響が大きい箇所

### 3-3. テスト対象外

次は単体テストの対象としない。

- Getter/Setterのみのクラス、単純な定数クラス、自動生成コード
- フレームワーク自体の動作確認
- 外部連携そのもの（API呼び出し、メール送信など。扱いは3-4を参照）

### 3-4. 外部連携

外部API呼び出し、メール送信、ファイル出力、バッチ実行などは、単体テストから直接実行しない。EasyMockで依存コンポーネントをモック化し、引数・呼び出し条件・例外時の挙動だけを検証する。

### 3-5. キャッシュ利用箇所（※保留）

一部の処理はキャッシュを利用しており、テスト前の初期化・ロードとテスト後のクリアが必要になる。ただし仕様が未確定のため、現時点ではテストコードにプレースホルダだけを置き、実装の詳細には踏み込まない。

```java
// TODO テストに必要なキャッシュを初期化・ロードする
// cacheTestSupport.loadRequiredCache();
```

初期化処理は共通のテストサポートクラスに集約し、個別テストには散らさない。

## 4. テストデータ作成方針

### 4-1. TestDataBuilder方式

- **目的**：EntityやDTOを1件生成する。業務的な意味をメソッド名で表し、デフォルト値を一箇所で管理する。
- **例**：`DbXxxTestDataBuilder` に `unapprovedXxx()` や `approvedXxx()` といった状態別メソッドを用意し、必要な差分だけ上書きする。
- **責務**：オブジェクト生成のみ。DB登録は行わない。

### 4-2. Fixture方式

- **目的**：1テーブル／1機能の単位でDB登録済みの状態を作り、テスト開始時の業務状態を簡潔に表す。
- **例**：`DbXxxFixture` が `DbXxxTestDataBuilder` でデータを組み立て、DaoやService経由でinsertする。
- **責務**：テーブル／機能単位に閉じたDB状態の構築。

### 4-3. Scenario Fixture方式

- **目的**：複数テーブルにまたがる業務シナリオの状態を作り、業務フローの前提条件をまとめる。
- **例**：請求書・明細・承認履歴・支払履歴を揃えて「承認済み請求書が存在する」状態にする。
- **注意**：肥大化しやすい。単純な1件生成や1テーブルだけの登録には使わない。

### 4-4. SQLファイル方式

- **用途**：H2用のDDL、固定マスタ、複数テーブルの共通初期状態をSQLで管理する。Gitで差分を追え、CIでも再現しやすい。
- **Excel由来データ**：既存のExcelマスタはSQLへ変換して取り込む。Excelを直接読む方式は採らない。
- **個別テストの業務データ**：大量データをSQLに書き下すのは避け、Builder／Fixture／Scenario Fixtureで用意する。

### 4-5. 方式別の責務一覧

| 方式 | 主な責務 | 例 |
| --- | --- | --- |
| TestDataBuilder | 1件のEntityやDTOを生成 | `DbXxx`、`XxxDto` |
| Fixture | 1テーブル/1機能単位でDB登録済み状態を作る | `DbXxx` をDB登録済みにする |
| Scenario Fixture | 複数テーブルをまたぐ業務シナリオ状態を作る | 承認済み請求書、支払確認済み状態など |
| SQLファイル | DDL、固定マスタ、複数テーブルの基本初期状態 | テーブル作成、マスタ投入 |

## 5. テストコード記述ルール

### 5-1. テストクラス命名

テスト対象クラスに対応させ、パッケージ構成も本番コードに合わせる。

- 例：`XxxLogic.java` → `XxxLogicTest.java`、`XxxService.java` → `XxxServiceTest.java`
- 配置は `src/test/java` 配下に統一し、業務コードとの対応を分かりやすくする。
- テスト支援クラス（Builder / Fixture / Scenario Fixture / CacheTestSupport / DatabaseTestSupport）は `testsupport` 配下にまとめる。

### 5-2. テストメソッド命名

既存規約を尊重しつつ、テスト対象・条件・期待結果が読み取れる名前を付ける。`test1`、`test2` のような連番や、中身の分からない名前は使わない。

例：

- `testApproveXxx_WhenStatusIsUnapproved_ThenStatusBecomesApproved`
- `testCalculateAmount_WhenTaxRateIsTenPercent_ThenReturnsTaxIncludedAmount`

### 5-3. Given / When / Then

各テストメソッドは次の3区分をコメントで区切り、テストコードがそのまま仕様の補助資料として読める構成にする。

- `// Given`：前提条件を用意する。TestDataBuilder・Fixture・Scenario Fixture・SQLでデータ状態を整える。
- `// When`：テスト対象メソッドを実行する。
- `// Then`：結果を検証する。戻り値、DB更新結果、状態遷移、例外、モックの呼び出し回数などを確認する。

### 5-4. assert

- JUnit標準のassert（`assertEquals`、`assertTrue`、`assertFalse`、`assertNull` など）を使う。
- AssertJやHamcrestといった外部ライブラリは導入しない。
- ステータス、金額、更新件数、例外条件など、業務上重要な状態は明示的に検証する。

### 5-5. 例外系テスト

- 例外の発生だけを確認する場合は `@Test(expected = XxxException.class)` を使う。
- メッセージやエラーコード、例外後の状態まで見たい場合は `try-catch` 内で `fail()` を使って検証する。

### 5-6. EasyMock

- 新規テストはEasyMockに統一する。既存にMockitoが残っていても追加はせず、EasyMockへ寄せていく。
- Logic層では Dao や外部依存を EasyMock で差し替え、条件分岐と例外処理を重点的に見る。
- Service層でも、外部連携やキャッシュなど対象外の依存は EasyMock で差し替える。

## 6. H2インメモリDB利用方針

### 6-1. DDLと固定マスタの管理

- H2用のDDLと固定マスタはSQLファイルで管理し、Gitで差分をレビューできるようにする。
- マスタはExcelに頼らず、SQLへ変換して固定化する。

### 6-2. DB初期化方式

テストごとにクリーンなDB状態を保つことが前提となる。S2JUnitは各テストメソッドをトランザクション内で実行し、終了時に自動でロールバックする。本方針ではこの標準挙動を基本とし、テストのたびにDBを初期状態へ戻す。

ロールバックでは戻しきれないケース（コミットを伴う処理や、複数トランザクションをまたぐ検証など）に限り、次の方式を補助的に併用する。

- **テスト前のDELETE/TRUNCATE**：対象テーブルを実行前にクリアする。
- **テストクラス単位でのH2再作成**：クラスごとにDBを作り直す。

### 6-3. テスト独立性

- テストは実行順序によらず、単体でも再現できること。
- あるテストが他のテストのDB状態・キャッシュ状態に影響を与えないこと。
- CI環境でも同じ結果が再現できること。

### 6-4. H2と本番DBの差異

H2はデータ型や方言が本番DBと異なることがある。SQL構文やデータ型に依存するロジックは特に注意し、できる限り標準SQLの範囲で書く。H2特有の挙動に寄りかかったテストは避ける。

## 7. CI組み込み方針

### 7-1. 単体テスト実行

- CIで単体テストを自動実行し、実行順序や環境差による不安定さがないことを確認する。
- ローカルPC固有のパスや設定に依存させない。
- H2のDDL・固定マスタ・初期データはCI上でも再現できるようにする。
- 外部API・メール・ファイル連携などの外部依存はモック化し、CIでも完結させる。

### 7-2. カバレッジ計測

カバレッジはCIで計測し、プロジェクト全体・パッケージ単位・重要クラス単位で確認する。重要な業務ロジックに抜けがないかを見るのが狙いで、数値そのものを目標にはしない。

### 7-3. 静的解析

導入初期は任意、または段階導入とし、運用が落ち着いてから品質チェック項目に加える。バグ検出、複雑度、コーディング規約違反などが候補となる。

### 7-4. 品質ゲートの前段としての利用

CIの単体テストとカバレッジを品質ゲートの前段に位置づけ、次の観点でレビューする。

- 単体テストがすべて成功している
- 重要な業務ロジックに十分なテストがある
- カバレッジが極端に下がっていない
- AI生成テストがCI上でも再現性をもって動く
- 静的解析の結果が許容範囲に収まっている

## 8. 導入ステップ

一気に全体へ広げず、段階的に進める。おおまかな流れは次のとおり。

1. 試験導入の対象機能を選ぶ。
2. 対象機能の Service / Logic / Dao / Entity / dicon / SQL / キャッシュ利用有無を洗い出し、範囲を確定する。
3. H2用のDDL SQLと固定マスタSQLを整える。
4. TestDataBuilderを作り、1件生成の雛形を用意する。
5. Fixtureを作り、テーブル単位のDB登録済み状態を準備する。
6. 必要に応じてScenario Fixtureを作り、複数テーブルにまたがる状態を準備する。
7. Logic層のテストを書く。原則はEasyMockでの差し替えとし、必要に応じて3-1のとおりH2を併用する。
8. 必要に応じてService層・Dao層のテストを書く。Service層はトランザクション境界とDB更新まで、Dao層はH2でSQLの正しさを確認する。
9. CIで単体テストとカバレッジ計測を回し、安定して通ることを確かめる。
10. 結果を方針書やAI向けプロンプトに反映し、チームでフィードバックを共有する。
11. 次の重要機能へ横展開する。テストパターンと支援クラスを再利用し、効率よく範囲を広げる。

## 9. AI向けテストコード生成プロンプト標準

AIエージェントにテスト生成を依頼するときは、以下の指示を含めたプロンプトを使う。クラス名や実装の詳細が不明なときは推測させず、確認事項として挙げさせる。

### 9-1. 基本指示

- テスト対象クラス名に対応する `対象クラス名Test` を生成すること。
- JUnit4形式（`@RunWith(Seasar2.class)`）を使用すること。
- Service/Logic/Dao のレイヤー構成に従い、モック化の要否を判断すること。
- トランザクション境界はService層を想定すること。
- モックライブラリは EasyMock を使用し、Mockitoは追加しないこと。
- assertはJUnit標準メソッドを用いること。
- テストメソッド名は条件と期待結果が分かるように命名すること（`testXxx_WhenCondition_ThenResult` 形式）。
- テストメソッド内は `// Given`、`// When`、`// Then` のコメントで区切ること。
- テストデータ作成には TestDataBuilder / Fixture / Scenario Fixture / SQL ファイルを適切に使い分けること。Excel方式は使わないこと。
- キャッシュ初期化・ロードは詳細を実装せず、プレースホルダコメントに留めること。（※保留）
- 外部連携コンポーネントはモック化し、呼び出し条件や引数、例外処理を検証すること。
- プロダクションコードは一切変更しないこと。テストしにくい構造でも本体コードを修正せず、モックやFixtureで対応すること。
- 不明点があれば推測で実装せず、「確認事項」として列挙すること。

### 9-2. プロンプト例

以下はAIエージェントにそのまま渡せるプロンプトの例。対象クラスやメソッドの名前、テスト条件などは案件に応じて置き換える。

```text
以下の条件でテストコードを生成してください。

* テスト対象クラス: XxxLogic
* テスト対象メソッド: approveXxx
* 業務仕様:
    - 未承認状態で承認すると、ステータスが承認済みに変わり、承認日時が現在日時で登録される。
    - 既に承認済みの場合は例外 XxxAlreadyApprovedException を送出する。

生成方針は以下のルールに従ってください。

1. JUnit4形式 (`@RunWith(Seasar2.class)`) を用いてください。
2. Daoや外部コンポーネントはEasyMockでモック化し、Service層のトランザクション境界を
   考慮する必要がある場合はH2を利用してください。
3. テストメソッド名は
   `testApproveXxx_WhenStatusIsUnapproved_ThenStatusBecomesApproved` のように
   条件と期待結果が分かる名前にしてください。
4. TestDataBuilder / Fixture / Scenario Fixture を使って前提データを用意してください。
   Excelは使用しません。
5. 各テストメソッドは `// Given`、`// When`、`// Then` の順で記述してください。
6. assertはJUnit標準のメソッドを使用してください。
7. キャッシュ初期化が必要な場合は以下のプレースホルダを記述し、実装には踏み込まないで
   ください。
   `// TODO テストに必要なキャッシュを初期化・ロードする`
8. 外部API呼び出しやメール送信などはモック化し、呼び出し条件と例外処理を検証してください。
   直接呼び出さないでください。
9. プロダクションコードを変更しないでください。テストしにくい部分があれば確認事項として
   列挙してください。
```

このプロンプトをテンプレートとして利用し、対象クラスや業務仕様に応じて具体的な記述を補う。

## 10. 確認事項

本方針書では決め切れない事項や、プロジェクト側で検討が必要な事項を以下に挙げる。

| 確認事項 | 説明 |
| --- | --- |
| 試験導入対象機能 | 最初にテスト導入する機能の選定。業務影響、不具合履歴、テスト容易性を基準とする。 |
| dicon詳細 | Seasar2のdicon設定やBean定義の詳細。テストで注入すべきコンポーネントの特定に必要。 |
| キャッシュ詳細 | キャッシュの利用箇所やロード手順。テスト初期化処理の設計にあたり、人間が補足する。 |
| CIツール詳細 | Jenkins、GitHub ActionsなどCIツールの種類と設定。DDL適用やカバレッジ計測の設定を検討する。 |
| カバレッジ計測ツール詳細 | JaCoCoなど具体的なツールの選定。CI上で結果を可視化する方法を決める。 |
| 静的解析ツール詳細 | Checkstyle、PMD、FindBugsなど導入時の選定と閾値。段階導入と品質ゲート連携の方法を検討する。 |

## おわりに

この方針書は、完璧なテスト体制を一度に求めるものではなく、まず動かせる形にすることを優先してまとめた。対象を絞って試し、回ったやり方を次の機能へ広げていく――その前提で読んでほしい。

運用を始めれば、ここで決め切れなかった点（10章）や、想定と違う挙動は必ず出てくる。そのときは現場の判断で方針側を更新し、このドキュメントを最新の状態に保ってほしい。テストコードとAI向けプロンプトを、チームの共通言語として一緒に育てていければと考えている。
